## TL;DR

- **What happened:** alabamapublicnotices.com migrated its CAPTCHA provider from Google reCAPTCHA v2 to Cloudflare Turnstile sometime between 06-30 and 07-01. Our pipeline was sending reCAPTCHA tokens to a Turnstile-validated form — silently rejected every time.

- **Nothing structural is broken:** foreclosure filter, ZIP-tier gate, address parser, DataSift two-pass upload, seen_ids dedup — all working. The pipeline was just stuck at the front door because the CAPTCHA solver was submitting wrong-provider tokens.

## Root cause — Cloudflare Turnstile migration

### Evidence

Discovered via headed-mode probe on 2026-07-02 07:22 UTC. When I inspected the page HTML after CAPTCHA:

```html
<script src="https://challenges.cloudflare.com/turnstile/v0/api.js" async="" defer="">
<div class="cf-turnstile" data-sitekey="0x4AAAAAADs-xpiU_yrMy8ZY" id="recaptcha">
<input type="hidden" name="cf-turnstile-response" id="cf-chl-widget-vtaml_response">
```

Note:
- `challenges.cloudflare.com/turnstile/v0/api.js` — Cloudflare's Turnstile API script
- `class="cf-turnstile"` and `cf-turnstile-response` — Cloudflare form field naming
- **`id="recaptcha"` is stale** — the div ID was kept for legacy compat but the actual widget is Turnstile
- **No `g-recaptcha-response` element on the page** — Google's reCAPTCHA field is completely gone

The old scraper was calling `solver.recaptcha()` on 2Captcha (returns a Google reCAPTCHA token) and injecting that token into `g-recaptcha-response` (element doesn't exist). The token was silently rejected server-side because the site is validating `cf-turnstile-response`.

### Why the intermediate fixes made things worse (not better)

Both v1 and v2 fixed the WRONG problem:

- **v1** thought the issue was that `wait_for_load_state("networkidle")` timed out. Replaced with a 10s selector wait. Added a "gate cleared" fallback: if the OLD reCAPTCHA gate text ("You must complete the reCAPTCHA") wasn't on the page, assume the gate was cleared. **The site had ALSO changed this message to "You must complete the CAPTCHA"** — so the old-text absence check kept firing → all 237 solves reported success with empty content.

- **v2** correctly noticed that empty content was being silently accepted. Added a hard requirement: `"Notice Content"` text must be visible after the wait. This CORRECTLY refused to claim success — but the reason content never appeared is because the token was still wrong-provider. 0 successes.

The lesson: without knowing the actual gate technology had changed, both fixes were guessing at symptoms.

## The v3 fix — what changed

### File: `src/captcha_solver.py`

Complete rewrite. Key changes:

1. **Sitekey extraction** — now dynamically read from `.cf-turnstile[data-sitekey]` attribute on each attempt (falls back to `config.TURNSTILE_SITEKEY_FALLBACK` if the widget hasn't hydrated). Robust against future sitekey rotation.

2. **Solver call** — `solver.turnstile(sitekey, url)` instead of `solver.recaptcha(sitekey, url)`. 2Captcha's Python library supports Turnstile natively; same $0.001-ish cost per solve.

3. **Token injection** — into `input[name="cf-turnstile-response"]` (not `g-recaptcha-response`). Dropped the entire `___grecaptcha_cfg` callback machinery that was reCAPTCHA-specific.

4. **Success detection** — two signals now, whichever appears first within 45s:
   - `body inner_text` length ≥ 1500 chars (site-format-agnostic, works even if the site removes the legacy marker)
   - Legacy `"Notice Content"` text present (post-migration verification via probe: **the marker IS present on successful Turnstile solves**, it was just never rendering when we were submitting wrong-provider tokens)

5. **Gate detection** — checks for BOTH the new message (`"You must complete the CAPTCHA"`) AND the legacy (`"You must complete the reCAPTCHA"`) to defensively survive a hypothetical rollback.

### File: `src/config.py`

Added `TURNSTILE_SITEKEY_FALLBACK = "0x4AAAAAADs-xpiU_yrMy8ZY"`. Kept `RECAPTCHA_SITEKEY` unused-but-present as a defensive default for the unlikely rollback case.

### File: `.github/workflows/daily-sweep.yml`

Bumped `timeout-minutes` from 360 → 480 (6h → 8h). Rationale in the "Known concerns" section below.

## Known concerns — Turnstile perf profile

### Observed on the headed probe (single-notice sample):

- Attempt 1: 79s, failed (`gate cleared but body text still short`)
- Attempt 2: 69s, failed (same)
- Attempt 3: 26s, **succeeded**
- Total: 173s per notice

### Extrapolation risk

Historical baseline reCAPTCHA solve time was ~30s/notice. If Wednesday's ~200-notice foreclosure spike takes 180s per notice:

- 200 notices × 180s = **10 hours** — well over the 6h cap → cancellations even on the new v3 fix
- Bumped workflow timeout to 8h (`ebaa090`) to buy headroom

### Attempt-1 failure mode

Attempts 1 and 2 on the probe returned tokens that seemingly got rejected by the site (page didn't advance despite gate-cleared appearance). Attempt 3 succeeded fast.

Hypotheses (unverified):
- **Session warmup** — first requests in a fresh Playwright session may be flagged by Cloudflare's Bot Management. Later requests in the same session (with cookies) may be trusted.
- **2Captcha token quality** — Turnstile tokens have a shorter validity window (~2 min) than reCAPTCHA v2 tokens (~2 min too, but Cloudflare may enforce it more strictly). Race condition between solve completion and submit.
- **Turnstile widget re-hydration** — the widget may reset its internal state on failure, and our next-attempt injection may hit stale DOM.

Investigation to do (after tomorrow's cron data comes in): grep the scrape log for the ratio of attempt-1 successes vs attempt-3 successes. If most notices succeed on attempt 1 in production (with a warmed session), 180s → ~30s and the timeout bump is comfortably enough.

### If production run cancels at 8h too:

Options in order of increasing effort:
1. **Reduce MAX_RETRIES from 3 → 2** — bounds worst-case time at 90s/notice
2. **Split scrape and upload into separate GHA jobs** — scrape job has 8h, upload job has 30 min; scrape can time out gracefully and hand off whatever it got
3. **Warm the session before batch scraping** — hit 1-2 notices before starting the queue, so attempts 1 in the main loop have the benefit of session warmup
4. **Switch to 2Captcha's newer Turnstile "callback" API** — a callback-style Turnstile solver that may have better attempt-1 success rates. Not yet investigated.
