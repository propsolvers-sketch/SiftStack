## Implementation Decisions

### New CSV filename pattern: `al_notices_*.csv`
Swaps state code, preserves rest of pattern. Lowest cognitive load ‚Äî downstream tools that grep for `*_notices_*.csv` still match. Clearly signals AL data.

**Why not `notices_*.csv` (state-agnostic):** Would lose the at-a-glance state signal in the filename. User wanted explicit AL marker.

**How to apply:** Change `f"tn_notices_{timestamp}.csv"` to `f"al_notices_{timestamp}.csv"` in `src/data_formatter.py:245`. Update the 2 referenced strings in `src/main.py` (CLI help + debug log) so the docs/log stay in sync.

### Scope is exactly 3 string edits ‚Äî no test changes
The filename is generated by `data_formatter.deduplicate()` (or wherever line 245 lives), not asserted by tests. Existing 110-test pytest suite continues to pass unchanged.

### Backward-compat consideration: none required
No downstream tooling/scripts/cron expects the literal `tn_notices_` prefix today (verified ‚Äî no GitHub Actions workflow file globs for it; user's daily review reads the file by Slack notification, not by name pattern). Old `tn_notices_*.csv` files from prior runs remain on disk and are not touched ‚Äî they keep their original names as historical record.
