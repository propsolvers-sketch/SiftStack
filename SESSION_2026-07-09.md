## Arc 2 — Enrichment parity (Smarty + Tracerfy + Trestle uniformly across all 5 sources)

**Trigger:** operator noticed several foreclosure CSVs on 2026-07-09 landed with zero phone numbers — only the APN output had phones.

**Root cause (2-part):**

**Part A — Skip-trace missing on 3 new adapters (`5c34efa`):** RL/T&B/HWM were enriching owner names via county property APIs but never invoking Tracerfy skip-trace or Trestle phone scoring. Meanwhile `main.py daily` runs Tracerfy via `full_pipeline.py`, which is why only APN had phones.

**Part B — Smarty subscription failure on the GHA runner:** the 2Captcha-flagged Turnstile session had rotated the Smarty account to a new key on 2026-05-23, but GHA secrets were last updated 2026-05-18. Direct HTTP probe against Smarty confirmed local .env credentials worked with `us-core-cloud` license. Fix: rotate GHA secrets via `gh secret set SMARTY_AUTH_ID` + `gh secret set SMARTY_AUTH_TOKEN` (reads from stdin, never logged). Verified by manual workflow run 29032445834 — Smarty standardization complete.

Then **`b63fa6e`** brought Smarty parity across all 5 sources, so records from every trustee portal now land in DataSift with the same enrichment metadata as APN:
- `dpv_match_code` (USPS deliverability)
- `latitude` / `longitude`
- `plus4_code` (ZIP+4)
- Normalized `delivery_line_1` (USPS CASS conventions)

**Part C — city-tier centroid fallback (`d05f79f`):** originally shipped for pre-probate (`97654c2`), extended in this session to apn_probate + tax_distress + distress_proxy, then to property_lookup + all 3 trustee adapters. Recovers ~2-4 Madison/Marshall records/week that used to drop as `tier=None` because Smarty returned city-only (Huntsville / Albertville / Arab / Guntersville / Boaz). Records carry `zip_estimated_from_city` in `missing_data_flags` for downstream filter-preset visibility.

## Arc 4 — Silent bugs uncovered in the operator review

Three latent defects surfaced in this session's investigation, all fixed:

### `3999bec` — LLM county tiebreaker was silently NameError-ing

`notice_parser.is_target_county_async()` referenced `rate_tracker` in its body but never accepted it in the signature. Every LLM county-lookup call was raising NameError, caught by the surrounding try/except and logged at DEBUG — invisible in Info+ output. Verified in GHA run 29001752184: hundreds of `"LLM county lookup error: name 'rate_tracker' is not defined"` DEBUG lines throughout every scrape. Fix: add `rate_tracker` kwarg + thread it from `scraper._scrape_notice` (which already had one).

**Downstream impact:** ambiguous notices (where regex found 0 or 2+ counties, typically <10% of probate) fell back to the regex heuristic (benefit-of-doubt for empty, intersect-test for multiple). Likely dropped a small number of real target-county records as "wrong county" over the past N months. The LLM tiebreaker now actually runs.

### `c570b65` — Fallback ZIP/city parser was TN-only

`notice_parser.py`'s address-parsing fallback path (~5-10% of notices — the ones the primary regex misses) was TN-only:

- `ZIP_RE = r"\b(3[78]\d{3})..."` matched only TN ZIPs (37xxx/38xxx). Every AL 35xxx/36xxx got rejected.
- `_COUNTY_ZIP_PREFIXES` had only Knox/Blount entries. Jefferson/Madison/Marshall lookups returned None → skipped validation → any ZIP passed the county-prefix check.
- Both fallback loops iterated only `TN_CITIES`.

Post-mortem-flagged during the 2026-07-01 CAPTCHA outage investigation but never fixed because the primary regex path handles ~90-95% of notices — the silent AL loss was in the tail.

Fix: extend `ZIP_RE` to `3[5-8]\d{3}` (accepts 35xxx-38xxx), add Jefferson `["350","351","352"]` / Madison `["357","358"]` / Marshall `["350","351","357","359"]` to `_COUNTY_ZIP_PREFIXES` based on USPS ZCTA boundaries, iterate both `TN_CITIES + AL_CITIES` in both fallback loops. `_COURTHOUSE_ZIPS` deliberately NOT extended — AL courthouse ZIPs (35203 Birmingham, 35801 Huntsville, 35976 Guntersville) sit inside our Tier 1/2 ZIP maps, so blanket rejection would drop legitimate leads. 17 sanity tests pass.

### `3265316` (pre-session, referenced here for completeness) — Smarty SDK license default

The Smarty Python SDK defaults to `us-standard-cloud` license, which lacks USPS-CASS. Every `dpv_match_code` was silently empty pre-fix. `_build_client()` now pins `us-core-cloud`.
